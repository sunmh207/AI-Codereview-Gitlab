# Code Review Expert Skill — 接入指南

## 做什么用的

这个 Skill 让 Claude Code 用资深工程师的视角帮你做 Code Review。

直接跟 AI 说"帮我看看代码有没有问题"，它也会给你反馈，但质量参差不齐——跟让一个新人做 review 一样，不是他不想认真看，是不知道该看什么、怎么看、按什么优先级来。

这个 Skill 解决的就是这个问题：给模型一套结构化的 review 框架，告诉它该检查什么、怎么分级、用什么格式输出。在这个基础上，我们又把团队的《软件设计开发自查清单》整合了进来，补充了数据库迁移、业务逻辑、消息驱动、接口设计等方面的检查规则。

---

## 审查流程

整个 review 按照真实的 Code Review 习惯编排，从宏观到微观，逐步递进。

### 第一步：了解改动范围

先跑 git diff 看改了哪些文件、多少行，判断变更类型（数据库脚本、业务逻辑、MQ、接口、定时任务）。如果改动超过 500 行，会先按模块分批 review，不会一口气全看完给一堆混在一起的反馈。

### 第二步：数据库 / Flyway 迁移检查

检查 migration 脚本是否安全：已发布脚本是否被修改了、SQL 是否幂等（DDL 有没有用 IF NOT EXISTS、DML 回填有没有保护条件）、大表 DDL 会不会锁表。这些问题轻则服务起不来，重则线上数据损坏。

### 第三步：核心业务逻辑检查

事务方法里 catch 了异常有没有重新抛出、热点资源有没有加锁、关键接口有没有做幂等、状态机有没有死胡同。这些是业务代码中最常见也最容易忽略的坑。

### 第四步：消息驱动检查

重点查一个高频问题：有没有在 @Transactional 方法里直接发 MQ 消息。事务回滚但消息已经发出去了，下游已经开始干活了，数据永久不一致。另外还会检查消费端的幂等处理和消息顺序性。

### 第五步：接口设计检查

列表接口有没有限制每页条数（没限制的话一个大查询就 OOM）、有没有 N+1 查询、更新接口是不是全量覆盖没有版本校验、批量操作有没有用批量接口。

### 第六步：定时任务检查

有没有防重入机制（上一轮没跑完下一轮又开始了会雪崩）、多实例部署有没有加分布式锁。

### 第七步：SOLID + 架构检查

这部分配了一份详细的 SOLID checklist，每个原则对应具体的"坏味道"。比如检查单一职责，不是泛泛地说"这个文件职责太多"，而是问一个具体的问题："这个模块有几个不同的修改理由？"如果一个文件既管 HTTP 请求、又管数据库操作、还管业务逻辑，那它大概率违反了 SRP。

### 第八步：发现可以删掉的代码

找出项目里的死代码——feature flag 关掉的、废弃的 API、没人用的工具函数。区分"可以直接删"和"需要制定计划再删"两种情况，给出具体的清理步骤。

### 第九步：安全扫描

XSS、SQL 注入、SSRF、路径穿越、密钥泄露都会检查。其中竞态条件这块写得特别详细，因为这是 review 时最容易忽略的——专门覆盖了共享状态访问、TOCTOU（检查后使用）、数据库并发、分布式系统四个子类，每个子类下面都有具体的代码模式。

### 第十步：代码质量扫描

异常处理有没有吞掉异常、N+1 查询、空值边界、除零保护、off-by-one。这些"小问题"在生产环境里都可能变成大事故。

### 最后：结构化输出 + 确认后修复

所有发现按严重程度分成 P0 到 P3 四个等级。输出报告后，Skill 不会自作主张改代码，而是先问你：要修全部、还是只修 P0/P1、还是修指定的几项。

这个"先 review 再确认"的设计是刻意的——Code Review 的价值不只是发现问题，更重要的是让你理解问题。如果 AI 直接帮你改了，你连有什么问题都不知道，这个 review 就没意义了。

## 检查项一览

### P0 — 必须修改，不改不能合入

标准：影响服务启动，或造成不可逆的数据损坏。

| 分类 | 检查项 | 后果 |
|------|--------|------|
| 数据库 / Flyway | 修改了已发布的 migration 脚本 | Checksum 校验失败，服务无法启动 |
| 数据库 / Flyway | SQL 不幂等：DDL 没用 `IF NOT EXISTS` 或项目工具；DML 回填没加保护条件 | 手动重跑产生重复数据或报错 |
| 数据库 / Flyway | 大表（>500万行或 >2GB）直接执行 ALTER TABLE | 长时间锁表，关联业务全部阻塞 |
| 核心业务 | @Transactional 方法中 catch 异常后未重新抛出 | Spring 认为正常返回，提交脏数据，无任何报错 |
| 消息驱动 | 在 @Transactional 方法内直接发送 MQ 消息 | 事务回滚时消息已发出，数据永久不一致 |
| 安全 | SQL 注入、认证绕过等安全漏洞 | 系统被攻击，数据泄露 |

### P1 — 应该修改，线上运行时会出问题

标准：不影响启动，但运行时会导致数据错乱、服务崩溃或流程卡死。

| 分类 | 检查项 | 后果 |
|------|--------|------|
| 核心业务 | 热点资源没有加锁 | 并发场景下数据错乱 |
| 核心业务 | 关键接口没有做幂等 | 重复调用产生重复数据 |
| 核心业务 | 状态机存在无出口的状态 | 单据卡死，无法继续流转 |
| 消息驱动 | 消费端没有处理重复消息 | 重投导致业务重复执行 |
| 接口设计 | 列表接口没有限制每页条数 | 大查询导致 OOM |
| 接口设计 | 高频接口存在 N+1 查询 | 数据量增长后超时 |
| 接口设计 | 更新接口全量覆盖，没有版本号校验 | 并发编辑互相覆盖 |
| 定时任务 | 没有防重入机制 | 任务堆积雪崩 |
| 定时任务 | 多实例没有加分布式锁 | 同一任务多节点同时执行 |

### P2 — 建议修改，本 PR 或后续 PR 处理

标准：长期存在会带来质量隐患，但短期不会引发事故。

| 分类 | 检查项 |
|------|--------|
| 数据库 / Flyway | 脚本命名不符合版本号规范 |
| 数据库 / Flyway | 新表缺少 create_time / update_time 审计字段 |
| 数据库 / Flyway | 启动时通过 JavaMigration 执行数据清洗 |
| 核心业务 | 该存快照的字段存成了引用（如下单价格只存了商品 ID） |
| 核心业务 | 关键业务规则未冗余存储（如换算率、批次号标记） |
| 消息驱动 | 顺序敏感的流程未使用一致的 Partition Key |
| 消息驱动 | 跨服务写操作使用同步调用 |
| 接口设计 | 低频路径存在 N+1 查询 |
| 接口设计 | 循环内逐条 insert/update |
| 数据指标 | 聚合计算未处理 NULL 和除零 |
| 代码结构 | SOLID 原则违反 |

### P3 — 可选改进

| 分类 | 检查项 |
|------|--------|
| 接口设计 | 详情页未拆分加载 |
| 代码 | 命名风格不统一 |
| 代码 | 非关键路径的轻微性能优化 |

## 设计思路

### 按需加载，不浪费上下文

所有检查规则分散在 `references/` 目录下的 8 个文件里（数据库、业务逻辑、消息、接口、安全、SOLID、代码质量、清理计划），加起来有几千字。如果全部塞进主文件，一上来就会占掉大量上下文窗口。

现在的做法是：`SKILL.md` 里只在需要的步骤写"加载 references/xxx.md"。模型执行到那个步骤时才会去读对应的文件，用完就释放。比如你这次的改动只涉及数据库脚本，它就只加载 `database-checklist.md`，不会把安全检查和 SOLID 检查的内容也读进来。

### 流程有节奏，从宏观到微观

试过把所有检查点平铺在一起，效果很差——模型会把安全问题和命名规范混在一起说，没有层次。

最终按照真实 Code Review 的习惯来编排：先看改动范围和类型，再看数据库和业务逻辑层面的设计问题，然后看安全，最后看代码质量细节。每一步之间是递进关系。好的 reviewer 不会上来就抠细节，而是先理解整体改动的意图和影响范围。

### 检查项有代码示例，不是空喊口号

每个检查项在 reference 文件中都配了具体的代码——错误写法长什么样、正确写法长什么样、review 时该问什么问题。不是那种"注意安全问题"之类的泛泛建议。

比如安全检查里的竞态条件，列了四个子类（共享状态访问、TOCTOU、数据库并发、分布式系统），每个下面都有具体的代码模式。比如业务检查里的事务吞异常，直接给了 @Transactional + try-catch 不抛出的代码片段和修复后的对比。

模型拿到这些具体的模式去匹配代码，输出质量会比泛泛的指令好很多。

### 先 review 再动手

Skill 输出报告后不会自动改代码，而是先列出所有发现，让你选择怎么处理。这个设计是刻意的——review 的价值不只是发现问题和修复问题，还在于让开发者理解问题。

## 使用方式

在 Claude Code 中执行：

```
/code-review-expert
```

或直接说：

```
帮我 review 一下当前的 git 变更
```

输出示例：

```
## Code Review Summary

Files reviewed: 5 files, 187 lines changed
Change type: Business Logic / Messaging
Overall assessment: REQUEST_CHANGES

### P0 - Critical
1. OrderService.java:45 — 事务内直接发送 MQ 消息
   - mqProducer.send() 在事务提交前就已发出
   - 修复：移到 @TransactionalEventListener(phase = AFTER_COMMIT)

### P1 - High
2. OrderController.java:78 — createOrder 接口缺少幂等设计
   - 重复 POST 会创建重复订单
   - 修复：orderNo 加唯一约束，或使用 Token 防重

## Next Steps
1. Fix all — 修复所有问题
2. Fix P0/P1 only — 只修复 P0 和 P1
3. Fix specific items — 指定修复某几项
4. No changes — 仅审查，不修改
```

## 如何接入其他项目

### 安装

将 `.skills/code-review-expert/` 目录复制到项目根目录下，不需要额外配置：

```
项目根目录/
└── .skills/
    └── code-review-expert/
        ├── SKILL.md                    # 主入口，定义审查流程
        ├── agents/
        │   └── agent.yaml
        └── references/                 # 详细检查规则，按领域拆分
            ├── database-checklist.md   # 数据库 / Flyway / SQL 幂等
            ├── business-checklist.md   # 事务安全、并发、幂等、状态机
            ├── messaging-checklist.md  # MQ 发送时机、消费幂等、顺序
            ├── api-design-checklist.md # 分页、N+1、批量、定时任务
            ├── solid-checklist.md      # SOLID 原则
            ├── security-checklist.md   # 安全（含竞态条件四个子类）
            ├── code-quality-checklist.md # 异常处理、性能、边界条件
            └── removal-plan.md         # 废弃代码清理模板
```

### 定制

**添加检查项：** 编辑 `references/` 下对应的 md 文件，按已有格式添加（标题标注级别、说明、错误代码、正确代码、review 时该问的问题）。同时在 `SKILL.md` 对应步骤中补充一行摘要。

**调整级别：** 修改 reference 文件中的 `[P0]` / `[P1]` 标记和 `SKILL.md` 中的对应标注。判断标准：

- **P0**：起不来，或者数据坏了没法恢复
- **P1**：跑着跑着会出事（崩溃、数据乱、流程卡死）
- **P2**：长期有隐患，短期不会出事
- **P3**：改了更好，不改也行

**适配技术栈：** 检查规则本身与框架无关（事务内不能发消息、SQL 要幂等等）。只需要修改 reference 文件中的代码示例。当前默认基于 Spring Boot + MyBatis + MySQL + RocketMQ/RabbitMQ。

## 常见问题

**与人工 Code Review 是什么关系？**
互补。Skill 负责规则性检查——清单上列出的已知问题。业务逻辑是否正确、需求理解是否到位、架构设计是否合理，这些仍然需要人来判断。

**改动量很大时会不会漏查？**
不会。超过 500 行的 diff 会先按文件列出变更概况，再按模块分批检查。

**出现误报怎么处理？**
选择 "No changes" 跳过。如果同一类误报反复出现，在 reference 文件中调整判定条件或增加排除规则。

**能接 CI 流水线吗？**
目前是交互式使用。CI 集成需要配合 Claude API 的非交互模式，技术上可行，可以作为后续方向。
